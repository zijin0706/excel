# Excel 数据匹配工具 —— 架构设计

---

## 一、设计目标

- 支持千万级数据量的 Excel/CSV 匹配
- 匹配规则动态可配置（YAML），无需改代码
- 自动拆分输出文件，确保 Excel 能正常打开
- 跨平台运行（Mac / Windows）

---

## 二、技术选型

| 组件 | 选型 | 原因 |
|------|------|------|
| 计算引擎 | DuckDB | 列式存储 + 向量化执行，千万行 JOIN 秒级完成，内存占用低 |
| 配置语言 | YAML | 可读性强，财务人员能看懂和修改 |
| Excel 读写 | openpyxl + pandas | 成熟稳定，支持 xlsx 格式 |
| 运行环境 | Python 3.9+ | 跨平台，安装简单 |

### 为什么选 DuckDB 而不是 Pandas？

| 对比 | Pandas | DuckDB |
|------|--------|--------|
| 1000 万行 JOIN | OOM / 数分钟 | ~10 秒 |
| 内存占用 | 全量加载 | 列式压缩 + 流式 |
| SQL 支持 | 无 | 完整 SQL |
| 跨文件查询 | 需手动合并 | 原生支持 |

## 三、整体架构

```
                        config.yaml
                            │
                            ▼
┌──────────────────────────────────────────┐
│              ConfigParser                  │
│  解析 YAML → PipelineConfig（数据模型）     │
│  校验：引用完整性、文件存在性                │
└──────────────────────────────────────────┘
                            │
                            ▼
┌──────────────────────────────────────────┐
│             SQLGenerator                   │
│  PipelineConfig → DuckDB SQL 语句列表      │
│  · source → CREATE TABLE AS SELECT ...    │
│  · from   → CREATE TABLE + filter         │
│  · join   → INNER/LEFT/RIGHT/NOT EXISTS   │
│  · sql    → 用户自定义 SQL                 │
└──────────────────────────────────────────┘
                            │
                            ▼
┌──────────────────────────────────────────┐
│               Executor                     │
│  · DuckDB 连接管理                         │
│  · 逐条执行 SQL，生成临时表                  │
│  · 流式分页读取（OFFSET/LIMIT）             │
│  · 支持内存溢出 → 磁盘临时目录               │
└──────────────────────────────────────────┘
                            │
                            ▼
┌──────────────────────────────────────────┐
│             OutputWriter                   │
│  · 每 chunk_size 行写一个文件               │
│  · 自动编号：prefix_001.xlsx               │
│  · 空结果输出带表头的空文件                  │
└──────────────────────────────────────────┘
                            │
                            ▼
                 data/output/
                 ├── matched/
                 └── unmatched/
```

## 四、模块说明

### 4.1 config/models.py —— 数据模型

```python
PipelineConfig
├── version: int                    # 配置版本号
├── settings: Settings              # 全局设置
│   ├── input_dir / output_dir      # 输入输出目录
│   ├── chunk_size: 300000          # 分片行数
│   └── duckdb: DuckDbSettings      # 内存/线程/溢出
├── sources: list[SourceConfig]     # 数据源列表
│   └── name, files, format, ...
├── pipeline: list[PipelineStep]    # 处理步骤（有序）
│   ├── from_ + filter              # 加载过滤
│   ├── join (JoinConfig)           # 表关联
│   └── sql                         # 自定义 SQL
└── outputs: list[OutputConfig]     # 输出定义
    └── from_, path, prefix, format
```

### 4.2 config/parser.py —— 配置解析

- `parse(path)` → `PipelineConfig`
- 校验规则：
  - source 名称唯一
  - pipeline 步骤名唯一
  - 所有引用（from/left/right/output.from_）指向存在的表
  - glob 文件匹配至少一个文件
- YAML 键名陷阱处理：`on` → `keys`（YAML 1.1 将 `on` 解析为布尔 `true`）

### 4.3 engine/sql_generator.py —— SQL 生成

| Step 类型 | 生成的 SQL |
|-----------|-----------|
| `source` | `CREATE TABLE x AS SELECT * FROM read_csv_auto([...])` |
| `from + filter` | `CREATE TABLE s AS SELECT ... FROM x WHERE ...` |
| `join inner` | `CREATE TABLE s AS SELECT * FROM L INNER JOIN R ON ...` |
| `join anti_left` | `CREATE TABLE s AS SELECT L.* FROM L WHERE NOT EXISTS (SELECT 1 FROM R WHERE ...)` |
| `sql` | `CREATE TABLE s AS {用户 SQL}` |

### 4.4 engine/executor.py —— 执行引擎

- `execute_batch(statements)` 逐条执行
- `stream_table(table, batch_size)` 分页流式读取
- 使用 `OFFSET/LIMIT` 分页，每次只加载一批到 pandas DataFrame
- 连接管理支持 `__enter__` / `__exit__`

### 4.5 output/writer.py —— 输出写入

- 流式消费 executor 的分批数据
- 每批写一个文件，命名 `{prefix}_{序号:03d}.xlsx`
- 空结果写一个带表头的空文件

## 五、处理流程

```
Step 1: CREATE TABLE table1 AS SELECT * FROM read_csv_auto('table1.csv')
Step 2: CREATE TABLE table2 AS SELECT * FROM read_csv_auto('table2.csv')
Step 3: CREATE TABLE all_data AS SELECT * FROM table1 UNION ALL SELECT * FROM table2
Step 4: CREATE TABLE aggregated AS SELECT ... GROUP BY ...
Step 5: CREATE TABLE matched AS SELECT * FROM aggregated WHERE 实际 = 应有
Step 6: CREATE TABLE unmatched AS SELECT * FROM aggregated WHERE 实际 != 应有
Step 7: 流式读取 matched → 分片写 Excel
Step 8: 流式读取 unmatched → 分片写 Excel
```

所有中间表存储在 DuckDB 内存中，不会落盘（除非设置 temp_directory）。

## 六、扩展能力

### 新增匹配类型

在 `JoinType` 枚举中新增类型，在 `_join_step` 中增加对应的 SQL 生成逻辑即可。

### 新增文件格式

在 `FileFormat` 枚举中新增，在 `_load_source` 中增加读取逻辑。DuckDB 原生支持 CSV、Parquet、JSON。

### 新增 Pipeline Step 类型

在 `PipelineStep` 中增加字段，在 `generate()` 中增加判断分支。

## 七、性能数据

| 数据量 | 操作 | 耗时 | 内存 |
|--------|------|------|------|
| 1000 万 × 2 表 INNER JOIN | 单列匹配 | ~10-20 秒 | ~2 GB |
| 1000 万行 GROUP BY + 聚合 | 10 个分组 | ~5-10 秒 | ~1 GB |
| 5000 万行 UNION ALL | 合并 | ~15-30 秒 | ~3 GB |

测试环境：Mac M1，8 核，16GB RAM。

---

## 八、安全与容错

- 配置文件校验在 SQL 执行前完成，错误提前暴露
- SQL 执行失败会打印失败语句前 300 字符
- 输出目录自动创建
- 重复列名 DuckDB 自动处理（加后缀 `_1`）
- 空结果不报错，输出带表头的空文件
